# Device Gateway 诊断 Agent 面试说明文档

## 1. 项目一句话介绍

这是一个面向设备网关场景的“诊断型 Agent”服务。

它的核心目标不是替代 Prometheus/Grafana，而是在现有监控体系之上，补足“解释能力”和“诊断能力”：

- 流量波动到底是正常业务波动，还是异常
- 告警出现后，可能原因是什么
- 指标异常能不能被日志和设备连接状态解释
- 面对一个设备或 topic，应该优先看哪些证据

我把它设计成一个独立的 `ops-agent-service`，使用 `Python + FastAPI + LangGraph + Pydantic` 实现，对外提供 HTTP API 和 Swagger，内部通过 LangGraph 编排指标、日志、网关状态的诊断流程。

---

## 2. 为什么要做这个项目

### 2.1 业务背景

原项目本身已经有两个核心能力：

- `Control Project`：负责设备 TCP/WebSocket 控制和连接转发
- `Exporter project`：负责将设备数据加工成 Prometheus 指标

也就是说，项目已经具备了“通信能力”和“观测能力”，但是还缺少“诊断能力”。

Prometheus 和 Grafana 很擅长：

 采集指标、画图、告警
但它们默认不擅长回答下面这些更接近业务排障的问题：

- 这个波动是不是正常的高峰期波动
- 这是某个设备掉线，还是整体链路抖动
- 是 topic 某一维异常，还是全局异常
- 有没有日志可以支撑这个判断
- 下一步该先看哪里

所以这个项目的价值就是把“可观测性”往前推进一步，从“发现异常”升级到“解释异常”。

---

## 3. 为什么不直接用 Grafana 自带能力

我不是要替代 Grafana，而是在 Grafana 之上补一层项目专属诊断能力。

原因有三个：

1. `Prometheus` 本身没有 AI Agent 能力。
   它的 `Agent Mode` 是采集转发模式，不是智能分析。
2. `Grafana` 的 AI/Assistant 能力更多集中在 `Grafana Cloud` 体系，而我当前是自建 `Prometheus + Grafana`。
3. 即使 Grafana 有通用 AI 能力，它也不天然理解这个项目里的领域语义，比如：
   - 哪个设备在线
   - 哪个连接是控制链路，哪个是推送链路
   - 哪种 topic 波动在业务上算正常
   - 某类错误日志对网关场景意味着什么

---

## 4. 为什么技术栈这样选

## 4.1 为什么不用 Java，改用 Python

虽然原项目主链路是 Java/Spring Boot，但我把 Agent 服务独立成 Python 服务，原因主要有四点：

1. `Python` 的 Agent 生态更成熟
   - LangGraph
   - LangChain
   - OpenAI 生态
   - Pydantic
2. 做指标、日志和规则分析时，Python 更灵活
3. Agent 服务本身是旁路诊断系统，不在实时转发主链路上，不必强绑定 Java 技术栈
4. 面试角度上，Python + LangGraph 的辨识度更高，也更容易讲清楚“AI 工程能力”

### 4.2 为什么选 FastAPI

我选 `FastAPI` 的原因：

- 天然适合做 HTTP API
- 自动生成 Swagger 文档，便于面试演示
- 和 Pydantic 配合很好，输入输出模型很清晰
- 对 Agent 服务这种“API-first”场景很合适

### 4.3 为什么选 LangGraph

这个问题很关键。

我不是单纯用 prompt + function calling，而是选了 `LangGraph`，因为这个场景本质上是一个“有状态、有步骤、需要可控编排”的诊断流程。

LangGraph 的优势：

- 能把诊断拆成明确节点
- 每个节点职责清晰，便于解释
- 适合后续扩展并行 investigator
- 支持 checkpoint / thread state，便于审计和恢复
- 比一大段 prompt 更可控，也更工程化

### 4.4 为什么没有用更重的多 Agent 方案

第一版我刻意没有做成复杂多 Agent 系统，是因为：

- 业务需求还没有复杂到必须多 Agent 协作
- 先解决“能稳定做诊断”比“架构看起来很炫”更重要

所以第一版我只用了 LangGraph 的必要能力：

- `StateGraph`
- tool calling
- structured output
- checkpoint / thread state

这也是一个很好的面试回答点：**我选了有扩展性的框架，但只使用当前业务真正需要的部分。**

### 4.5 为什么用 Pydantic

`Pydantic` 在这个项目里不是简单做参数校验，而是承担了三层作用：

1. API 输入输出模型定义
2. LangGraph 节点间状态结构定义
3. 诊断结果的结构化输出约束

这样可以避免大模型输出漂移，也方便后续接前端或接告警系统。

### 4.6 为什么没有采用标准 RAG

这个项目没有采用“分片 + embedding + 向量召回”的标准 RAG 流程，主要是因为当前场景更偏实时诊断，而不是静态知识问答。

核心原因有三点：

1. 当前最关键的证据不是知识文档，而是实时数据源
   - Prometheus 指标
   - Loki 日志
   - Gateway 连接状态
2. 这类问题天然带有明确过滤条件
   - 设备 ID
   - topic ID
   - 时间窗口
   - 错误关键词
   这种情况下，直接查 Loki / Prometheus 比先做 embedding 再召回更直接，也更稳定。
3. 目前项目里还没有足够成熟的静态运维知识库
   - 比如 SOP
   - 故障案例库
   - 运维手册
   - FAQ
   如果后续补齐这些资料，再叠加一层标准 RAG 会更合适。

所以这个项目更准确的说法是：**没有采用标准文档型 RAG，但保留了 retrieval-augmented 的核心思想。**

面试里可以这样说：

“这个项目没有采用经典 RAG 的分片、embedding 和向量召回方案，因为当前场景更偏实时排障，核心证据来自 Prometheus、Loki 和网关状态接口。不过它保留了 retrieval-augmented 的核心思想，也就是先检索外部证据，再基于证据做诊断，而不是让模型裸推理。”

### 4.7 为什么用 SQLite

第一版用 `SQLite` 是一个有意识的取舍：

- 部署轻
- 演示方便
- 不依赖额外中间件
- 足够支持 run 记录、审计、回放

如果后续进生产，可以平滑迁移到 PostgreSQL。

## 5. 整体架构设计

## 5.1 架构目标

我在设计时有两个核心原则：

1. 不影响原有网关实时通信主链路
2. Agent 必须是只读的、可审计的、可降级的

### 5.2 架构拆分

整个方案分成两层：

#### 第一层：原有网关与观测层

- `Control Project`
  - 设备 TCP/WebSocket 通信
  - 设备连接状态管理
  - 补充只读运维接口
- `Exporter project`
  - 输出 Prometheus 指标
- `Prometheus`
  - 存储和查询指标
- `Loki`
  - 存储和查询日志
- `Grafana`
  - 展示和告警

#### 第二层：诊断 Agent 层

- `ops-agent-service`
  - 接收诊断请求
  - 通过 LangGraph 编排诊断流程
  - 调用 Prometheus/Loki/网关运维接口
  - 生成结构化诊断结果
  - 保存每次分析记录

### 5.3 为什么单独做成服务

我没有把 Agent 逻辑直接嵌进 `Control Project`，而是拆成独立服务，原因是：

- 避免 AI 相关依赖污染主链路
- 避免模型调用影响实时控制延迟
- 便于单独扩展、部署、迭代
- 让控制链路和诊断链路解耦

一个好的回答是：

“实时控制系统追求稳定和低延迟，而 Agent 诊断是旁路增强能力，应该通过架构隔离来降低耦合和风险。”

## 6. 实现方案

## 6.1 对外接口设计

目前实现了三个主要接口：

### `POST /agent/analyze`

用于通用诊断。

输入：

- `question`
- `time_range`
- `device_id`
- `topic_id`

输出：

- `summary`
- `is_anomalous`
- `confidence`
- `suspected_causes`
- `evidence`
- `next_steps`
- `normalized_query`
- `assessment`

### `POST /agent/explain-alert`

用于解释 Grafana 告警。

输入：

- `labels`
- `annotations`
- `starts_at`
- `time_range`

这个接口会先把告警信息转成自然语言问题，再进入同一个 LangGraph 工作流。

### `GET /agent/health`

检查四个外部依赖的健康状态：

- LLM
- Prometheus
- Loki
- gateway ops API
  
---

## 6.2 LangGraph 工作流设计

我把图设计成一个“弱 agent 化”的单图多节点工作流。

节点如下：

### 1. `query_understanding`

职责：

- 解析用户问题
- 提取 `device_id` / `topic_id`
- 生成标准化查询参数
- 组装 PromQL / LogQL

这里既可以用 LLM 做轻量理解，也有 fallback 正则解析，避免完全依赖模型。

### 2. `fetch_metrics`

职责：

- 调用 Prometheus
- 读取当前时间窗指标
- 读取前一天 baseline 指标
- 解析成统一的 `MetricsEvidence`

### 3. `fetch_logs`

职责：

- 调用 Loki
- 查询同时间窗相关日志
- 按错误关键词提取关键日志
- 生成 `LogsEvidence`

### 4. `fetch_gateway_context`

职责：

- 调用网关只读 ops API
- 获取设备在线状态
- 获取最近事件
- 获取整体概览

### 5. `anomaly_evaluator`

职责：

- 优先进行非 LLM 规则判断
- 用指标偏离、日志匹配、连接状态综合判断异常性
- 输出 `AnomalyAssessment`

这是我刻意设计的“防幻觉层”。

### 6. `diagnosis_writer`

职责：

- 让模型基于已经拿到的证据做最终总结
- 生成结构化 JSON
- 如果模型不可用，则走 deterministic fallback

---

## 6.3 为什么先规则判断，再让 LLM 写结论

这是面试里非常加分的设计点。

我的思路是：

- LLM 擅长总结和解释
- 但不适合直接做底层事实判断

所以我把职责拆开：

- 规则层负责“事实判断”
- LLM 层负责“自然语言归纳和解释”

在当前实现里，`anomaly_evaluator` 这一步不是直接让模型决定 “是不是异常”，而是先跑一层确定性的规则计算。规则层大致分成四步：

1. 先从证据里提取信号
   - 从 `MetricsEvidence` 里取当前窗口均值、最新值、前一天同窗口 baseline 均值
   - 再计算 `delta_ratio = (current_mean - baseline_mean) / baseline_mean`
   - 从 `GatewayOpsEvidence` 里看设备是否离线
   - 从 `LogsEvidence` 里看同时间窗是否存在相关日志

2. 再根据这些信号命中规则
   - 如果流量相对前一天同窗口偏移超过阈值，就记一条 reason
   - 如果当前流量直接掉到 0，而 baseline 不是 0，也记一条 reason
   - 如果同时间窗查到了相关日志，再加一条 reason
   - 如果网关显示设备离线，再加一条 reason

3. 然后形成结构化判断
   - 根据命中的 reasons 和信号组合，得出 `is_anomalous`
   - 这里的 `reasons` 不是最终结论，而是“当前系统为什么怀疑异常”的依据列表

4. 最后再算 `confidence`
   - 这里的 `confidence` 不是模型概率，也不是机器学习意义上的置信度
   - 它更像一个工程上的“证据强度分”
   - 例如流量偏移越大、日志佐证越多、设备离线信号越强，分数就越高；如果某些证据源降级缺失，分数就会下调

`reasons` 在整个链路里有两个作用：

- 一方面它用于支撑规则层自己的判定，让 `is_anomalous` 不是一个黑盒布尔值
- 另一方面它会继续传给后面的 `diagnosis_writer` 或 deterministic fallback，作为 `suspected_causes` 和解释文本的输入

查日志在这里也不是为了“只要有日志就是异常”，而是为了给指标变化找佐证。也就是说，指标负责发现波动，日志和设备状态负责增强解释力与可信度。

这样做的好处：

- 减少幻觉
- 让判断更稳定
- 便于测试
- 更容易向面试官解释“为什么可信”

你可以这样回答：

“我没有让大模型直接决定是不是异常，而是让它基于规则计算后的事实和证据来写诊断结论，这样更可控。”

## 7. 数据来源与工具层设计

## 7.1 Prometheus

Prometheus 提供的是时间序列证据。

当前设计里主要查询：

- 当前窗口流量
- 前一天同窗口 baseline
- total/flow 相关指标

目的不是做复杂 forecasting，而是先解决最有解释价值的对比：

- 当前 vs 历史
- 当前 vs baseline

## 7.2 Loki

Loki 提供的是日志证据。

查询策略上我没有全量暴力扫描，而是先按关键词过滤，比如：

- `error`
- `disconnect`
- `timeout`
- `parse failure`
- `exception`

这样既控制成本，也更契合排障场景。

## 7.3 Gateway Ops API

这是我自己补的一层项目专属上下文。

新增的接口有：

- `/ops/devices`
- `/ops/devices/{deviceName}/status`
- `/ops/events`

这层的价值在于，Prometheus 和 Loki 只能告诉你“发生了什么”，但网关运维接口可以告诉你“设备当前是什么状态”。

它把领域知识补进来了。

## 8. 落地细节

## 8.1 只读原则

整个 Agent 第一版坚持只读，不做设备控制。

原因：

- 降低风险
- 避免大模型误操作
- 更适合面试展示
- 先把诊断做好，再考虑闭环控制

### 8.2 降级设计

这个项目里我非常强调“降级”。

例如：

- Prometheus 不可用时，不能伪造指标结论
- Loki 不可用时，要明确说日志证据缺失
- LLM 不可用时，要回退到 deterministic diagnosis

所以当前实现里：

- LLM 有 fallback
- `health` 接口能看出依赖状态
- `/agent/analyze` 在证据源缺失时仍能返回结构化结果，但会降低置信度



### 8.3 审计与回放

每次分析都会落到 SQLite，记录：

- 请求类型
- 原始请求
- 标准化查询
- 各类 evidence
- assessment
- final answer
- 状态与耗时

这样做的价值：

- 便于调试
- 便于面试演示
- 为后续人工审核和回放打基础

### 8.4 结构化输出

我没有让接口返回自由文本，而是返回结构化 JSON。

这样做的好处：

- 前端和其他系统更容易接
- 输出更稳定
- 便于自动测试
- 便于后续做审批流或工单联动

## 10. 我做这个项目时的关键取舍

这一段特别适合面试。

### 10.1 取舍一：不用复杂多 Agent

因为第一版更重要的是稳定和可讲清楚，而不是“看起来很炫”。

### 10.2 取舍二：不用向量库/RAG

因为当前问题的核心不是文档问答，而是 observability 诊断。
真正关键的数据源是：指标、日志、连接状态而不是知识库。

更准确一点说，这里不是标准文档型 RAG，而是“实时数据源驱动的检索增强诊断”。
我保留了先检索证据、再生成结论的思路，但没有引入分片、embedding 和向量召回，因为当前并没有成熟的运维知识库，核心价值也不在静态文档问答。

### 10.3 取舍三：先 SQLite，不急着 PostgreSQL

因为 v1 更强调演示和最小可用，而不是大规模生产化。

### 10.4 取舍四：先做解释，不做自动执行

因为诊断正确性尚未充分验证前，自动控制风险很高。

## 11. 这个项目的亮点

如果面试官问“你觉得这个项目最大的亮点是什么”，可以从下面几个角度答。

- 规则优先
- 结构化输出
- 只读权限
- 降级策略
- run 审计

### 亮点五：工程化闭环完整

不是只有模型调用，还包括：

- API
- 数据源接入

- 测试

## 12. 面试官高频追问与参考回答

## 12.1 为什么一定要用 LangGraph，而不是直接 function calling？

参考回答：

“如果只是一次简单工具调用，function calling 就够了。但这个场景是一个有步骤的诊断流程：先理解问题，再取指标，再取日志，再取设备状态，再做规则评估，最后生成结论。LangGraph 能把这些步骤显式表达出来，也方便后续做 checkpoint、审计和扩展。”

## 12.2 为什么不是多 Agent，而是单图工作流？

参考回答：

“第一版问题复杂度还没有高到必须多 Agent 协作。先用单图工作流把证据链和诊断链打通，等后续真的需要并行 investigator 或人工审核节点时，再扩展成多 Agent 架构更合理。”

## 12.3 你这个流程看起来还是一条链，为什么说是 LangGraph？

参考回答：

“这个问题问得很对。我当前这版并不是复杂决策图，而是图框架里的轻量工作流。主链很明确，但中间不是纯线性串行，而是先做 query understanding，然后把指标、日志、网关上下文三个证据获取节点扇出，再汇合到 anomaly evaluator 和 diagnosis writer。所以它已经具备图结构特征，只是复杂度是刻意控制过的。也正因为如此，我能在不把系统做重的前提下，用 LangGraph 把状态流转、节点职责和后续扩展空间表达清楚。”

如果面试官继续追问，可以补一句：

“更准确地说，我这个项目当前是弱图结构，不是复杂图，但它已经不是单条 chain 了。”

## 12.4 你怎么控制大模型幻觉？

参考回答：

“我没有让大模型直接决定异常，而是先做规则评估，再把事实和证据交给模型做总结。其次，输出被 Pydantic 结构约束。最后，如果证据缺失，系统会明确标记 degraded，而不是编造结论。”

## 12.5 为什么要单独做 gateway ops API？

参考回答：

“Prometheus 和 Loki 提供的是通用 observability 视角，但网关场景还需要领域上下文，比如设备在线状态、最近连接/断开事件。这类信息放在专门的 ops API 里，可以让诊断更贴近真实业务。”

## 12.6 如果 Prometheus 或 Loki 挂了怎么办？

参考回答：

“系统会降级。健康检查会标出依赖异常，分析接口仍然会尽量返回已有证据下的结果，但会降低置信度，并明确说明哪些证据源缺失。”

## 12.7 为什么先用 SQLite？

参考回答：

“这是 v1 的工程取舍。它部署轻、演示方便、能满足 run 审计需求。等进入更正式环境，再迁移到 PostgreSQL。”

## 12.8 为什么不用 SQLite 直接做断点续传？

参考回答：

“SQLite 不是不能做 checkpoint 持久化，但我这个项目当前工作流比较短，v1 更强调审计和回放，所以我先把 SQLite 用在最终分析结果落盘上，而没有把它扩展成节点级断点续传存储。因为 checkpoint 写入会更频繁，对并发和状态恢复要求也更高。后续如果真要做服务重启后的精确续跑，我会更倾向用专门的持久化 checkpointer，或者 PostgreSQL 这类更适合服务场景的存储。”

如果面试官继续追问，可以补一句：

“当前版本更偏结果可回放，而不是中途可续跑。两者相关，但不是一回事。”

## 12.9 传统 workflow 也能做，为什么还要引入 LLM？

参考回答：

“传统 workflow 确实也能完成一部分能力，比如查指标、查日志、跑规则判断。但我引入 LLM 不是为了替代规则，而是为了补自然语言理解和证据归纳这两块传统流程比较僵硬的能力。比如用户会直接问‘帮我看看 device-204 最近 2 小时 topic-22 的流量是不是异常’，这里需要从自然语言里提取设备、topic 和分析重点；另外最终输出也不是简单规则命中，而是把指标、日志和设备状态组织成一段可读的诊断解释。异常判断本身仍然是规则优先，所以这是 workflow + LLM 的组合，不是纯模型驱动。”

如果面试官继续追问，可以补一句：

“我把 LLM 放在理解和解释的位置，而不是放在底层事实判断的位置，这样能兼顾灵活性和可控性。”

## 12.10 你最后用的模型是什么？

参考回答：

“当前默认配置的模型是 `gpt-4o-mini`，这是出于成本、速度和效果平衡做的选择。代码里模型名是可配置的，不是写死某个厂商；我这里走的是 OpenAI-compatible 接口，所以后续如果要切换兼容模型，改配置就可以，不需要重写业务流程。”

如果面试官继续追问，可以补一句：

“我当前更关注的是把诊断链路跑通，所以先选了一个小模型做 v1，后续如果场景变复杂，再按效果和成本调整模型档位。”

## 12.11 这个项目的 token 消耗大概是多少？

参考回答：

“当前版本没有把 token usage 做成精确的账单级监控，所以我一般会明确说这是工程估算，不是精确计费值。一次 `/agent/analyze` 通常最多两次模型调用：第一次是 query understanding，第二次是 diagnosis writing。第一步 token 很少，通常只是短 prompt 加问题文本；第二步会带上指标摘要、最多 5 条日志、最多 5 条网关事件和 assessment。粗略估算，单次诊断在降级模式下总 token 大概是 500 到 900；正常有真实日志时，大概是 800 到 2000；更高一点的情况通常出现在日志上下文比较满的时候。” 

如果面试官继续追问，可以补一句：

“我已经做了上下文裁剪，比如日志只取前 5 条、每条截断、事件也只取前几条，所以 token 消耗整体是可控的。”

## 12.12 这个项目支持多轮对话吗？

参考回答：

“这个项目更准确地说是支持轻量多轮，而不是开放式聊天记忆。因为它的核心不是闲聊，而是围绕同一个诊断问题持续追问，所以我保留了 `thread_id` 作为会话标识，用来串联同一条诊断链路。每次分析结果也会带上 `thread_id` 并落到 SQLite，便于后续回看同一问题的上下文。它强调的是诊断线程连续性，而不是维护一大段自由对话历史。”

如果面试官继续追问，可以补一句：

“这个设计更贴合排障场景，因为真正重要的是设备、topic、时间窗和证据链，而不是完整聊天记录。”

## 12.13 断点续传、记忆压缩这些点你是怎么考虑的？

参考回答：

“我在架构上已经按 LangGraph 的 checkpoint / thread state 思路设计了，但当前版本使用的是内存型 checkpointer，更偏开发和演示阶段的状态管理，还不是服务重启后的生产级断点续传。记忆压缩上我也没有做通用聊天摘要，而是做结构化压缩：只保留设备、topic、时间窗、标准化查询和证据摘要这些真正对诊断有价值的状态。因为这个项目本质上是排障工作流，不是开放域聊天，所以结构化状态比保留整段自然语言历史更稳定，也更适合审计和回放。”

如果面试官继续追问，可以补一句：

“当前版本更偏线程级状态管理和结果可回放，后续如果要做真正的持久化续跑，再把 checkpointer 落到更合适的持久化存储里。”

## 12.14 为什么这里用 thread_id，而不是 user_id？

参考回答：

“因为这个项目当前管理的是诊断线程，而不是用户关系，所以我更关心‘这几次请求是不是属于同一条排障链路’，而不是‘是谁发起的请求’。一个用户可能同时排查多个设备、多个 topic 或多个告警，如果只按 `user_id` 聚合，上下文很容易混在一起。`thread_id` 更像问题实例 ID，更符合排障场景，也更方便后续和告警、工单、审计链路对接。”

如果面试官继续追问，可以补一句：

“如果后续接入真正的登录体系，更合理的做法通常是同时保留 `user_id` 和 `thread_id`，前者表示是谁发起，后者表示是哪条诊断链路。”

我会优先做四件事：

## 13. 你可以怎么讲这个项目

### 简洁版

“我做了一个基于 LangGraph 的设备网关诊断 Agent。原项目已经有 Java 网关和 Prometheus 指标，但缺少对异常的解释能力，所以我单独做了一个 Python 服务，接入 Prometheus、Loki 和网关状态接口。它可以回答流量波动是否正常、告警背后的可能原因、指标异常能否被日志和设备状态解释。技术上我用了 FastAPI 提供 API 和 Swagger，用 LangGraph 显式编排诊断流程，用 Pydantic 约束结构化输出，并把每次分析写入 SQLite 以便审计和回放。这个项目的重点不是简单调用大模型，而是把 AI 放进真实可观测性链路里，并通过规则优先、降级设计和只读权限来控制风险。” 

### 偏技术版

“我把 Agent 设计成旁路诊断服务，而不是嵌进实时控制链路。这样做的原因是控制系统追求低延迟和稳定性，而 AI 推理本身具有不确定性。工作流上我没有直接用 prompt + function calling，而是用 LangGraph 把 query understanding、metrics fetch、logs fetch、gateway context fetch、anomaly evaluation、diagnosis writing 这些步骤显式拆开。规则判断先于 LLM，总结后于规则，这样能降低幻觉并增强可解释性。”

## 15. 总结

这个项目的本质不是“做了一个 AI 聊天接口”，而是：

- 基于真实网关业务场景
- 接入真实 observability 数据源
- 用 LangGraph 编排一个可控的诊断流程
- 用规则 + LLM 的组合方式平衡准确性和解释性
- 用只读、降级、审计来保证工程可用性
