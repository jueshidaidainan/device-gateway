# 模拟面试追问与参考回答

## 1. 你为什么会想到做这个项目？

我原来的项目已经把设备数据接进来了，也已经通过 Prometheus 和 Grafana 做监控和告警，但我发现它只能告诉我“有异常”，不能很好地告诉我“为什么异常”。而网关场景里很多问题需要结合指标、日志和设备连接状态一起看，所以我就做了一个诊断型 Agent，把这些证据串起来做解释。

---

## 2. 为什么不用 Grafana 原生能力，自己再做一个 Agent？

Grafana 更擅长通用 observability，尤其是展示和告警。但我这个项目有比较强的领域特征，比如设备在线状态、控制链路和推送链路的区别、topic 语义、网关日志模式，这些都是通用平台不天然理解的。所以我不是替代 Grafana，而是在 Grafana 之上补了一层项目专属诊断能力。

---

## 3. 为什么这个 Agent 要单独拆成服务，而不是直接写进原来的 Java 网关？

因为网关主链路是实时通信系统，它更强调低延迟和稳定性；而 Agent 分析涉及模型推理和多数据源查询，本身不适合放进实时转发链路里。拆成独立服务可以实现架构隔离，避免 AI 相关依赖和延迟影响原有主链路，也更方便独立部署和迭代。

---

## 4. 既然原项目是 Java，为什么 Agent 这块改成 Python？

主要是因为 Python 在 Agent 和数据处理生态上更成熟。像 `LangGraph`、模型 SDK、时序和日志处理这类能力，Python 的组合效率更高。另一方面，这个 Agent 本身就是旁路服务，不在主链路里，所以没有必要强行和 Java 栈绑定，独立成 Python 服务反而更灵活。

---

## 5. 为什么选 FastAPI？

我这个 Agent 是典型的 API-first 服务，FastAPI 很适合这种场景。它对 Pydantic 支持很好，接口定义清晰，而且能自动生成 Swagger 文档，面试演示和后续联调都很方便。

---

## 6. 为什么选 LangGraph，而不是直接用 prompt + function calling？

因为这个问题本质上不是一次性的问答，而是一个有状态、有步骤的诊断流程。我要先解析问题，再取指标、取日志、取设备状态，然后做规则判断，最后再生成结论。LangGraph 可以把这些步骤显式拆开，让流程更可控、更容易扩展，也更方便解释和调试。

---

## 7. 为什么没有做成多 Agent？

第一版没有必要上复杂多 Agent。当前问题更适合先把单图工作流和证据链跑通。多 Agent 会增加架构复杂度和调试成本，如果业务还没到那个复杂度，收益不高。我的做法是先选一个有扩展性的框架，但只用当前真正需要的能力。

---

## 8. 你在 LangGraph 里具体是怎么设计节点的？

我把它拆成 6 个节点：

1. `query_understanding`：解析自然语言问题，提取 `device_id`、`topic_id`、时间范围，生成查询参数
2. `fetch_metrics`：从 Prometheus 拉当前窗口和 baseline 指标
3. `fetch_logs`：从 Loki 拉对应时间段的关键日志
4. `fetch_gateway_context`：从网关只读接口拿设备在线状态和最近事件
5. `anomaly_evaluator`：先做规则判断
6. `diagnosis_writer`：让模型基于证据生成结构化诊断结果

这个设计的核心是把“数据采集”“规则判断”“语言总结”分开。

---

## 9. 你说“规则优先、LLM 后置”，具体是什么意思？

意思是我没有让模型直接判断是不是异常，而是先用确定性的规则去判断，比如当前指标和 baseline 的偏离、设备是否在线、有没有错误日志匹配。规则层先给出事实层判断，再让 LLM 去做总结和解释。这样能降低幻觉，让结果更稳定，也更容易测试。

---

## 10. 你具体用了哪些规则？

目前第一版是轻量规则，重点是实用性而不是复杂度：

- 当前流量和前一天同窗口 baseline 的偏离
- 当前 latest 值是否掉到接近 0
- 设备当前是否在线
- 同时间窗是否出现 disconnect / timeout / error 类日志

如果这些信号叠加出现，就会提高异常判断和置信度。

---

## 11. 为什么 baseline 用“前一天同窗口”，而不是更复杂的算法？

这是一个 v1 的工程取舍。前一天同窗口 baseline 的实现成本低、可解释性强，而且在很多业务场景里已经能捕捉比较明显的异常。后续如果数据更稳定、样本更多，可以继续扩展成周同比、移动平均、MAD 或更复杂的时序方法。

---

## 12. 为什么要接 Loki？

只看指标很容易知道“有波动”，但很难知道“为什么波动”。日志能提供更直接的因果线索，比如连接断开、超时、解析失败、异常堆栈等。所以 Loki 不是锦上添花，而是这个诊断链路里的关键证据源。

---

## 13. 为什么还要自己补 Gateway Ops API？

因为 Prometheus 和 Loki 是通用观测视角，但我这个项目还有一层领域上下文，比如设备当前在线不在线、最近有没有连接/断开事件，这些信息通用平台不一定有或者不够方便取。我补了一层只读运维接口，相当于把项目自己的语义接到了 Agent 上。

---

## 14. 这些接口为什么坚持做成只读？

这是出于安全和工程可控性的考虑。第一版的目标是诊断和解释，不是自动执行控制命令。模型推理有不确定性，如果一上来就给执行权限，风险比较高。先把只读诊断做稳，再考虑审批后执行，会更合理。

---

## 15. 如果 Prometheus、Loki 或 LLM 某一个挂了，系统怎么办？

系统会降级，而不是直接报废或者编造结论。

- 如果 Prometheus 挂了，就不能做可靠的指标判断，会明确提示 metrics unavailable
- 如果 Loki 挂了，会提示日志证据缺失，但如果还有设备状态，也可以做部分判断
- 如果 LLM 挂了，会走 deterministic fallback，至少返回规则判断和结构化总结

所以这个系统的设计目标不是“所有依赖必须全好才能工作”，而是“依赖缺失时仍然可控地退化”。

---

## 16. 为什么要做健康检查接口？

因为 Agent 这种服务比较依赖外部系统，健康检查接口能快速告诉我：

- 模型通不通
- Prometheus 通不通
- Loki 通不通
- 网关只读接口通不通

这不只是运维方便，也能体现整个系统是面向工程落地设计的。

---

## 17. 为什么要把每次分析结果写到 SQLite？

主要是为了审计和回放。Agent 的结果不是只看当下，还需要能回头看：

- 当时问了什么问题
- 系统取了哪些证据
- 规则层怎么判断的
- 最后模型怎么总结的

SQLite 是 v1 的低成本选择，部署简单，足够支持这个阶段的需求。

---

## 18. 为什么不用 PostgreSQL？

不是不能用，而是第一版没必要。SQLite 更适合快速落地和演示，而且已经能满足 run 记录、审计和回放。等后续需要多实例部署、更强并发和更复杂查询时，再迁移 PostgreSQL 更合理。

---

## 19. 你觉得这个项目最难的地方是什么？

我觉得最难的不是把接口搭出来，而是怎么让系统既有 AI 能力，又不失控。因为如果只是接模型，很容易做成一个看起来能回答问题、但实际上不可靠的 Demo。所以我比较强调规则优先、结构化输出、只读权限、降级和审计，这些是在工程上把它“收住”的关键。

---

## 20. 你觉得这个项目最有价值的设计点是什么？

我觉得最有价值的设计点有两个：

1. 把 Agent 作为旁路诊断服务，而不是嵌入主链路，保证系统解耦和安全
2. 把“规则判断”和“LLM 总结”拆开，前者负责事实，后者负责解释，兼顾准确性和表达能力

---

## 21. 如果让你继续迭代，你下一步会做什么？

我会优先做四件事：

1. 接入真实 Prometheus / Loki 数据，补强真实 case
2. 把 SQLite 升级到 PostgreSQL
3. 增加 run 查询接口和简单前端展示页
4. 在 LangGraph 里增加更复杂的 investigator 节点，比如 metrics investigator、logs investigator、connection investigator

---

## 22. 如果面试官问“你这个项目和普通 AI Demo 的区别是什么”，怎么回答？

我会说，这个项目最大的区别是它不是围绕“聊天”设计的，而是围绕“诊断”设计的。它接的是 Prometheus、Loki、网关状态接口这些真实系统数据源，不是单纯让大模型生成答案；同时我还做了规则判断、结构化输出、降级和审计，所以它更像一个工程系统，而不是一个套壳式 AI Demo。

---

## 23. 如果面试官问“你在这个项目里体现了什么工程能力”，怎么回答？

我觉得主要体现在这几个方面：

- 能从业务问题出发做架构拆分
- 能做技术选型，而不是追热点
- 能把 AI 能力接进真实系统，而不是停留在 Demo
- 能考虑只读、安全、降级、审计这些工程问题
- 能把工作流拆成清晰模块，便于后续维护和扩展

---

## 24. 最后一问：如果要一句话总结这个项目，你会怎么说？

这是一个把 AI 能力工程化地接入设备网关可观测性场景的诊断 Agent，通过 LangGraph 编排指标、日志和设备状态的证据链，以规则优先、LLM 后置的方式实现异常解释与告警诊断。
