# 简历项目描述

## 版本一：标准版

### 项目名称

基于 LangGraph 的设备网关诊断 Agent

### 项目描述

基于原有设备网关与可观测性体系，设计并实现一个只读诊断型 Agent 服务，接入 `Prometheus` 指标、`Loki` 日志和网关运行状态接口，支持分析流量波动是否异常、解释 Grafana 告警原因，并输出结构化诊断结果。

### 技术栈

`Python`、`FastAPI`、`LangGraph`、`Pydantic`、`Prometheus`、`Loki`、`Grafana`、`SQLite`

### 我的工作

- 独立设计诊断 Agent 的整体方案，将 AI 能力从实时控制主链路中解耦，落为独立 `ops-agent-service`
- 使用 `LangGraph` 设计状态化诊断流程，拆分 `query_understanding`、`fetch_metrics`、`fetch_logs`、`fetch_gateway_context`、`anomaly_evaluator`、`diagnosis_writer` 等节点
- 封装 `Prometheus`、`Loki` 和网关只读运维接口客户端，构建指标、日志、设备连接状态三类证据链
- 实现“规则优先、LLM 后置”的诊断策略，先通过 baseline 偏离、设备在线状态、日志匹配等规则做异常判断，再由模型生成结构化解释，降低幻觉风险
- 基于 `FastAPI + Pydantic` 提供 `/agent/analyze`、`/agent/explain-alert`、`/agent/health` 接口及 Swagger 文档
- 使用 `SQLite` 持久化每次分析的请求、证据、assessment 和最终结果，支持审计与回放
- 为原 Java 网关补充 `/ops/devices`、`/ops/devices/{id}/status`、`/ops/events` 等只读运维接口，提供设备在线状态和最近连接事件
- 完成 Conda 环境搭建、依赖安装、基础单元测试和服务级 smoke test

### 项目亮点

- 不是单纯接入大模型，而是将 AI 真正落地到设备网关可观测性诊断场景
- 使用 `LangGraph` 做显式工作流编排，具备较强的可解释性和可扩展性
- 通过只读权限、降级策略、结构化输出和审计存储保证工程可控性

## 版本二：压缩版

设计并实现基于 `LangGraph + FastAPI` 的设备网关诊断 Agent，接入 `Prometheus` 指标、`Loki` 日志和网关只读状态接口，支持流量异常分析和 Grafana 告警解释；采用“规则优先、LLM 后置”的方式降低幻觉风险，并通过 `Pydantic` 结构化输出与 `SQLite` 审计存储提升系统可控性与可追溯性。

---

## 版本三：更偏工程亮点版

围绕设备网关可观测性场景，搭建独立的诊断 Agent 服务，使用 `LangGraph` 编排指标、日志、设备连接状态的多步骤诊断流程，对外提供标准化 HTTP API 和 Swagger；通过补充网关运维接口、设计可降级诊断链路和 run 审计机制，将 AI 能力以工程化方式接入现有监控体系。

## 面试时简历项目描述怎么展开

如果面试官让你展开这一条项目经历，建议按下面顺序讲：

1. 原项目已经有网关和监控，但缺少“异常解释能力”
2. 所以做了独立的诊断 Agent，而不是嵌进主链路
3. 为什么用 `LangGraph`
4. 为什么是“规则优先、LLM 后置”
5. 最后讲工程可控性：只读、降级、审计、结构化输出
