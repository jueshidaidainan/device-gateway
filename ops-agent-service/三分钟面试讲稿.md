# Device Gateway 诊断 Agent 3 分钟讲稿

## 版本一：3 分钟完整表达

我这个项目的核心思路是，在原有设备网关和监控体系之上，加一层“诊断型 Agent”。

原项目本身已经有两个能力：一个是 `Control Project`，负责设备 TCP/WebSocket 通信和控制转发；另一个是 `Exporter project`，负责把设备数据转换成 Prometheus 指标。所以它已经能完成通信和监控，但缺少一个很关键的能力，就是对异常进行解释。

比如 Prometheus 和 Grafana 可以告诉我某个设备或者某个 topic 的流量出现了波动，也可以配告警，但它们不擅长回答更接近排障的问题，比如这个波动是不是正常业务高峰，是不是某个设备离线了，是不是链路抖动，或者有没有对应日志能支撑这个判断。

所以我做了一个独立的 `ops-agent-service`，技术栈用的是 `Python + FastAPI + LangGraph + Pydantic`。我没有把它嵌进原来的 Java 网关主链路，而是拆成旁路服务。这样做的原因是实时控制系统更看重稳定性和低延迟，而 Agent 分析和模型推理本身是相对重且有不确定性的，所以我通过架构隔离来降低风险和耦合。

这个 Agent 服务主要接三个数据源：第一是 `Prometheus`，用来拿时间序列指标；第二是 `Loki`，用来查同一时间窗口的日志；第三是我自己在网关里补的一层只读 `ops API`，用来拿设备在线状态和最近连接事件。这样我就把通用 observability 数据和项目自己的领域上下文结合起来了。

在流程编排上，我没有简单做成 prompt 加 function calling，而是用了 `LangGraph`。因为这个场景天然是一个多步骤、有状态的诊断流程。我把它拆成几个明确的节点：先做 `query_understanding`，解析问题并标准化查询；再分别做 `fetch_metrics`、`fetch_logs`、`fetch_gateway_context`；然后在 `anomaly_evaluator` 节点先做规则判断，比如看当前流量和历史 baseline 的偏离度；最后再进入 `diagnosis_writer`，让大模型基于这些证据生成结构化诊断结果。

这里我有一个比较强调的设计点，就是“规则优先，LLM 后置”。我没有让大模型直接拍脑袋判断是否异常，而是先用确定性的规则去判断，再让模型做总结和解释。这样可以减少幻觉，也让结果更稳定、更可解释。

对外我提供了三个接口：`/agent/analyze` 用于通用诊断，`/agent/explain-alert` 用于解释 Grafana 告警，`/agent/health` 用于检查模型、Prometheus、Loki 和网关状态接口的连通性。每次分析我还会把请求、证据、assessment 和最终结果写入 SQLite，便于审计和回放。

这个项目我觉得最有价值的地方，不是单纯接了一个大模型，而是把 AI 能力真正落进了可观测性链路里，并且通过只读权限、降级策略、结构化输出和审计存储，让它更像一个工程系统，而不是一个 Demo。

## 版本二：1 分钟精简表达

我做了一个基于 `LangGraph` 的设备网关诊断 Agent。原项目已经有 Java 网关和 Prometheus 指标，但缺少对异常的解释能力，所以我单独做了一个 Python 服务，接入 `Prometheus`、`Loki` 和网关只读状态接口。它可以回答流量波动是否正常、告警背后可能是什么原因、指标异常能否被日志和设备状态解释。技术上我用了 `FastAPI` 提供 API 和 Swagger，用 `LangGraph` 显式编排诊断流程，用 `Pydantic` 保证结构化输出，并把每次分析写入 `SQLite` 方便审计和回放。这个项目的重点不是简单调用大模型，而是把 AI 放进真实 observability 场景里，并通过规则优先、只读和降级设计保证可控性。

## 版本三：面试官问“你做的最有亮点的项目是什么”时的回答

我最近做的一个比较有代表性的项目，是给设备网关做了一个诊断型 Agent。它不是普通聊天机器人，而是接到了真实监控链路里。原项目已经能把设备数据转成 Prometheus 指标，也能通过 Grafana 展示和告警，但还缺少异常解释能力。所以我做了一个基于 LangGraph 的服务，把指标、日志和设备连接状态串起来，回答像“这个流量波动是否正常”“这个告警更像设备离线还是链路问题”这种问题。它的亮点在于我没有让大模型直接决定结论，而是先做规则判断，再让模型基于证据去总结，所以既有 AI 的表达能力，也保留了工程上的可控性。
